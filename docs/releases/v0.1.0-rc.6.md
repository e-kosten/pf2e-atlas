# v0.1.0-rc.6

## Summary

This release candidate adds the first local Atlas web-app slice on top of the
Rust retrieval stack from rc.5. The CLI still remains the primary packaged
surface, but Atlas now has a localhost web API, a React/Ant Design frontend
prototype for search and record detail workflows, and Rust-owned app contracts
that can also support future interactive surfaces. It also starts the Rust tag
corpus model and resolves more Foundry localization during ingest.

New and improved user-facing surfaces:

- `atlas web` starts a localhost API service for interactive Atlas workflows,
  with repo/global path overrides, automatic local port selection, and
  `--open` browser launch support.
- The new `atlas-web`, `atlas-app-service`, and `atlas-app-model` crates define
  the local web boundary: HTTP routing stays thin, interactive workflow state
  lives below the web transport, and TypeScript DTOs are generated from Rust
  app contracts.
- `web/atlas-ui` adds the first Vite/React frontend prototype for local search,
  result browsing, filter editing, and record detail inspection.
- The web UI uses readable search URLs, keeps selected filters discoverable,
  preserves result-window state, and improves pagination, sorting, and filter
  clearing behavior.
- Web filters are now driven by the backend filter editor model instead of a
  hand-coded frontend field catalog, including dynamic value discovery,
  collapsed optional filter groups, tri-state value picking, and metric
  comparison filters.
- Record presentation for the web surface now includes richer section, badge,
  relationship, table, inline, and content-block DTOs, plus friendlier creature
  size labels.
- Web request execution is bounded so local HTTP traffic cannot queue unlimited
  blocking retrieval work ahead of the app service.
- Foundry localization references are resolved during ingest, reducing exposed
  `@Localize[...]`-style text in normalized content and generated records.
- `atlas-tags` introduces the Rust-owned tag catalog, assignment, suggestion,
  YAML loading, and validation model for future authored retrieval tags.
- `atlas tags validate` validates a tag corpus under `data/tags` and can emit
  either terminal output or the standard JSON envelope.
- Architecture docs now describe the local web app boundary, app-service/app-model
  ownership, and the Rust tagging model. New backlog items capture remaining web
  polish, frontend serving, filter hardening, and tag follow-up work.

## Install Notes

- Install or update the CLI normally. For a published RC, `just release-install`
  can install the release matching the local `crates/atlas-cli` version, or a
  specific tag such as `just release-install v0.1.0-rc.6`.
- Run `atlas setup --check --json` after upgrading to inspect source,
  embedding, and artifact readiness.
- Run `atlas setup` if setup reports a missing, stale, or incompatible
  artifact. This rc includes ingest localization and presentation changes that
  are most visible after rebuilding the local SQLite index.
- `atlas web` starts the Rust localhost API service. In this rc, the checked-in
  frontend remains a developer prototype under `web/atlas-ui`; run it with Vite
  from the repository checkout and proxy `/api/*` to the `atlas web` URL.
- Frontend contributors should run `npm install` under `web/atlas-ui`, then use
  `npm run dev` for local development or `npm run verify` for the frontend gate.
- App DTO changes must keep the checked-in TypeScript bindings fresh. Regenerate
  them with `cargo test -p atlas-app-model export_typescript_bindings -- --ignored`
  and validate with `cargo test -p atlas-app-model`.
- Tag corpus authors can validate the default corpus path with
  `atlas tags validate`, or pass `--path data/tags --json` for machine-readable
  validation output.
