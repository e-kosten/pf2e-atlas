# v0.1.0-rc.5

## Summary

This release candidate tightens Atlas's Rust retrieval stack after the rc.4
CLI expansion. It keeps the same core command surface, but moves more artifact,
filter, record, and retrieval policy into their owning crates, improves local
index build performance, and makes the user-facing record/search vocabulary
more consistent.

New and improved user-facing surfaces:

- Search, similar-record lookup, README examples, and CLI guidance now use
  record `kind` terminology consistently instead of the older `family` wording.
- `atlas search` uses page-based pagination, so scripts can request later result
  pages directly with `--page` plus `--limit`.
- `atlas similar` has more focused validation, filtering, and explanation
  behavior, with seed resolution and candidate filtering handled by the search
  layer.
- Graph, remaster, variant, and similar lookups now resolve record references in
  the search layer, reducing duplicated CLI-side lookup behavior.
- Similar-record ranking batches outgoing reference evidence reads, improving
  lookup speed for records with many references.
- Variant group references are resolved during search result construction, so
  variant context is more complete in hydrated results.
- Rich content documents now preserve a stronger structured content model for
  record presentation, JSON projection, FTS text, and embedding text.
- Filter discovery and filter lowering now live with the search/index boundary
  that owns them, and filtered keyset composition is centralized for more
  consistent structured-filter behavior.
- SQLite artifact storage now uses Diesel migrations and a checked-in Diesel
  schema in `atlas-index`; artifact schema, validation, writing, discovery, and
  read policy are owned by one crate.
- Index discovery-catalog writes are faster, especially for generated field,
  value, and metric catalogs during local artifact builds.
- `atlas-index` internals are reorganized into explicit read, write, artifact,
  discovery, graph, record, search, and vector modules; the separate
  `atlas-artifact` and `atlas-discovery` crates were folded into `atlas-index`.
- CLI command parsing was split out of `main.rs` into command-specific argument
  modules, keeping `main.rs` focused on top-level execution.
- Runtime setup, path, retrieval, and error handling were split into focused
  `atlas-runtime` modules.
- Public crate APIs across `atlas-index`, `atlas-search`, `atlas-record`,
  `atlas-runtime`, and `atlas-sqlite-vec` were narrowed so downstream callers
  rely on intended product-facing entry points.
- The bundled `pf2e-atlas-cli` agent skill has updated guidance for `kind`
  filters, graph context, similar-record lookup, relationship filters, and JSON
  handling.
- New getting-started and CLI-example docs cover install, setup readiness,
  strict name resolution, search filters, pagination, graph context, similar
  records, filter discovery, and JSON envelopes.
- Workspace validation is quieter by default and supports
  `just verify --verbose` or `scripts/verify.sh --verbose` for full command
  output.
- A `just release-install` helper installs a published Atlas release or RC by
  tag, defaulting to the current CLI crate version.

## Install Notes

- Install or update the CLI normally. For a published RC, `just release-install`
  can install the release matching the local `crates/atlas-cli` version, or a
  specific tag such as `just release-install v0.1.0-rc.5`.
- Run `atlas setup --check --json` after upgrading to inspect source, embedding,
  and artifact readiness.
- Run `atlas setup` if setup reports a missing, stale, or incompatible artifact.
  This rc includes artifact schema, storage, content-model, and filter changes
  that can require rebuilding the local SQLite index.
- Existing scripts and examples should use `--kind` rather than `--family`.
- Existing scripts that page through search results should pass `--page` and
  `--limit` explicitly.
- Downstream Rust callers should import artifact schema, discovery, validation,
  and read/write functionality from `atlas-index`; the previous
  `atlas-artifact` and `atlas-discovery` workspace crates are gone.
- SQLite schema changes now belong in the Diesel migration under
  `crates/atlas-index/migrations/` plus the checked-in Diesel schema at
  `crates/atlas-index/src/schema.rs`.
